BEYOND THE MANDATE · WHAT AUDIT LOOKS LIKE AT RUNTIME

PHI never enters the LLM. Ever.

Compliance with the rule is the floor, not the ceiling. Authrex is built so that even a worst-case LLM prompt leak cannot expose protected health information — because the LLM never sees it. Hash-chained audit. Per-tenant row-level security. Synthetic-only demos. The trust posture is structural, not policy-based.

HIPAA · 45 CFR § 164

Minimum-necessary, redacted at the gate

The PHI Sanitizer runs before the first LLM token is spent. Patient name, MRN, DOB, address, phone — all stripped and replaced with stable hashed identifiers. Only patient initials propagate to the agent pipeline.

  • Direct identifiers replaced with deterministic SHA-256 hashes
  • Redaction receipts written to phi_redactions ledger
  • BAA-ready · runs entirely inside our VPC
  • No PHI in logs, prompts, observability, or LLM provider traces
SOC 2 TYPE II · TARGET

Per-tenant isolation, structurally enforced

Every database query carries a tenant claim from the JWT. PostgreSQL row-level security (RLS) policies enforce isolation at the kernel level — application bugs cannot leak data across tenants.

  • RLS policies on cases, decisions, agent_runs, audit_ledger
  • Service-account JWTs with 1-hour expiry · rotated keys
  • Per-tenant cost ledger · per-tenant rate limiting
  • Encryption at rest (KMS) · in transit (TLS 1.3) · in process (memory-only PHI)
DEMO POLICY · SYNTHETIC ONLY

We never demo on real patients

Every fixture in the demo cohort is generated by Synthea, the open-source synthetic patient simulator. The demo cases — HER2+ APPROVE, HER2+ REFER, HER2-negative DENY — model real oncology pathways but use entirely fictional patients.

  • 3 synthetic fixtures · all from backend/app/synthea/seeds/
  • FHIR R4 + USCDI v3 conformance verified per fixture
  • Real LLM pipeline · synthetic data only · zero PHI risk
  • Patient initials in slides are fictional (M.K. · R.S. · etc.)
SHA-256 · TAMPER-EVIDENT

Hash-chained append-only audit

Every event — case created, agent fired, verdict synthesized, decision composed — appends to audit_ledger with a hash that includes the previous row's hash. Modification of any past row breaks the chain detectably.

  • SHA-256 chain · prev_hash → curr_hash linked
  • Verifiable via SELECT verify_chain(case_id) in < 1 sec
  • Append-only — no UPDATE, no DELETE on this table ever
  • 10-year retention · CMS-0057-F § IV.A compliant

PHI data-flow — built so leakage is structurally impossible

Lane A · what enters the LLM agent pipeline
FHIR Bundle → PHI Sanitizer (BLOCK) → ClinicalSnapshot · initials only → 7 LLM agents · Sonnet 4.6 → Verdict + citations
Lane B · what enters the audit ledger
PHI Sanitizer → phi_redactions row · hashed before/after → audit_ledger row · SHA-256 chain
Lane C · what reaches a downstream payer (TriZetto)
Decision → TriZetto envelope · re-attached PHI · BAA-covered transit → Payer adjudication
PHI re-enters the flow only at Lane C, only inside the BAA-covered transit, only after the decision is fully cited. The LLM has already done its job using initials and FHIR pointers. The agent layer is structurally PHI-blind.

What Authrex never does (built into CLAUDE.md project conventions)

Never log PHI — even in synthetic mode. Patient initials only. Enforced by sanitizer + lint rule.
Never call the LLM without trace_agent — every invocation produces an audit row before the first token is generated.
Never fabricate clinical facts — agents fail closed if evidence is missing. AMBIGUOUS is a verdict, not an error.
Never delete from audit tables — append-only by DB constraint. Updates rejected. Deletions logged + halted.
Never hardcode secrets — KMS-managed keys. Rotated quarterly. Per-tenant scopes.
Never let the LLM own the verdict — verdict is a deterministic function of LLM-graded criteria. The reasoning is the explanation, not the decision.
